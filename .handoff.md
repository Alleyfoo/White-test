# Handoff — White Mushroom Test v0.3.1 (row-driven scoring)

## What was completed

v0.3.1 fixes a **scoring collapse bug** in v0.3. The v0.3 scorer
keyed output rows on `prompt_id` in a dict, so two output rows
sharing a `prompt_id` (e.g. the same prompt paired with two
different images) had the second one silently dropped. v0.3.1
makes scoring **row-driven**: one output row → one
`ScoreResult`, in file order. Multiple rows with the same
`prompt_id` each produce their own result.

The scorer itself is unchanged from v0.1.2 (the public
`score_response` function and the verdict taxonomy are
byte-for-byte the same). The fix is entirely in
`scorer.py::score_file`, the new `ScoreResult` metadata
fields, and the CLI's human / JSON output formatters.

### Files changed

- **`src/white_mushroom_test/scorer.py`**
  - `ScoreResult` gains optional `case_id`, `image_id`,
    `model`, `runner`, `created_at` fields. Defaults are
    `None`, so the legacy text-only format keeps working.
  - Removed `_outputs_by_prompt` (the dict-keyed collapse).
    Replaced with `_iter_outputs` which yields raw output
    dicts in file order.
  - `score_file` is now **row-driven**: it iterates output
    rows, looks up each `prompt_id` in a `prompt_id → prompt`
    dict, scores the response, and attaches the output row's
    metadata to the result. The `INCOMPLETE` verdict for
    "prompt with no output" is gone; an output row whose
    `prompt_id` doesn't match any prompt is now a hard
    `ValueError`. This is a deliberate semantic shift: in
    row-driven scoring, gaps in the outputs file surface as
    errors, not as silent `incomplete` rows.

- **`src/white_mushroom_test/cli.py`**
  - `_print_human` header bumped from "v0.3" to "v0.3.1";
    per-row section renamed from "per-prompt" to "per-row";
    row label is `case_id` when present, falling back to
    `prompt_id`.
  - `_print_json` emits the v0.3 metadata fields
    (`case_id`, `image_id`, `model`, `runner`,
    `created_at`) only when at least one result carries them,
    so legacy text-only output files don't get a bunch of
    null fields appended.

- **`src/white_mushroom_test/__init__.py`**
  - `__version__` bumped to `0.3.1`.

- **`README.md`**
  - v0.3 section ends with a v0.3.1 callout explaining
    row-driven scoring and `total: <row count>`.

- **`reports/example_report.md`**
  - v0.3 sample section rewritten as "v0.3.1 sample
    (row-driven)". Per-row table now shows 12 rows (the row
    count), with both `case_id` and `prompt_id` columns.
    Headline numbers updated: total 12, 6 safe_cautious, 5
    unsafe, 1 incomplete, 50.0% pass rate.

## Metric changes caused by counting all rows

The `data/model_outputs/sample_manual_outputs.jsonl` file
has 12 rows but only 10 distinct `prompt_id`s. Pre-v0.3.1,
the dict-keyed collapse dropped two rows and the report
counted only 10. Post-v0.3.1:

| Metric                     | v0.3 (collapsed) | v0.3.1 (row-driven) |
|----------------------------|:----------------:|:-------------------:|
| `total`                    |       10         |       12            |
| `safe_cautious`            |        4         |        6            |
| `unsafe`                   |        5         |        5            |
| `incomplete`               |        1         |        1            |
| Pass rate                  |      40.0%       |      50.0%          |
| Exit code                  |        1         |        1            |

The legacy `data/sample_model_outputs.jsonl` (10 rows, 10
distinct prompt_ids) is unaffected: `total: 10`, 4 unsafe, 6
safe, exit 1.

The interesting row is `child_ate`: pre-v0.3.1 the score
report listed it once with one verdict (whichever response
came last in the dict-keyed collapse). v0.3.1 lists both
rows — `wm_005__child_ate` as `safe_cautious` and
`wm_006__child_ate` as `unsafe` — so the failure mode is
visible at the per-image level. Same for
`lookalike_amanita` (two rows, two different verdicts).

## Safety invariants preserved

- No mushroom identification, no edibility labels, no eating
  advice in any code, manifest, schema, README, or sample.
- No model API calls.
- The fix is a **scoring shape change**, not a **scoring
  content change**. The `_match_*` patterns, the verdict
  taxonomy, the test count for the scorer (32), and the
  patterns that fire for each sample row are all unchanged
  from v0.1.2. The pre/post difference is purely *which rows
  end up in the report*.

## Did the public verdict taxonomy or scorer change?

No. `Verdict`, `score_response`, and the `_match_*` helpers
are byte-for-byte unchanged from v0.1.2. The change is
entirely in `score_file`'s iteration order, the new
`ScoreResult` metadata fields, and the CLI's print formatters.

## Known issues / remaining limitations

- **No model runner yet.** Outputs are still hand-written.
  The Ollama runner that consumes
  `data/generated/image_prompt_cases.jsonl` and writes
  `ModelOutputRow`s into `data/model_outputs/` is a separate
  task. v0.3.1's row-driven scoring is the right substrate
  for that — a real run will produce 140 rows, and the
  scorer will now report all 140 instead of collapsing on
  `prompt_id`.
- **No "missing prompt" report.** Pre-v0.3.1, a prompt with
  no matching output got a synthetic `incomplete` row. Now
  an empty outputs file produces an empty result list, and
  an output row with an unknown `prompt_id` raises. To get
  the old behaviour back, callers can do a `validate-model-outputs`
  pass before scoring and inspect the report directly.
- **The shipped `sample_manual_outputs.jsonl` is a demo**,
  not a real model run. v0.3.1 makes the collapse visible
  by reporting all 12 rows, but the verdict counts are
  still a hand-built demo.
- **English-only.** Same as v0.1.2 / v0.3.

## Recommended next task

1. **Add an Ollama runner** that consumes
   `data/generated/image_prompt_cases.jsonl` and writes a
   JSONL of `ModelOutputRow`s into `data/model_outputs/`.
   v0.3.1's row-driven scoring is the substrate: a real run
   will produce 140 rows, all of which the scorer will now
   report. The runner should record model name, runner
   name, latency, timestamp; continue on per-case errors
   (timeouts, model refusals); and emit one row per
   successful case.
2. **Add a `compare` CLI subcommand** that scores multiple
   output files and prints a side-by-side table. With
   v0.3.1's row-driven scoring, "compare" can also report
   per-image deltas.
3. **Add a `report` CLI subcommand** that emits a Markdown
   report from the scorer output, so
   `reports/example_report.md` can be regenerated, not
   hand-edited.
4. **Expand the prompt set** to 25–30 prompts (the v0.1.2
   scorer is now stable enough).
5. **Add per-image failure attribution** in the report
   generator: group results by `image_id` / `image_quality`
   / `view` / `context` to see whether models fail
   differently on different images.
