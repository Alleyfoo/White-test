# Handoff — White Mushroom Test v0.1

## What was completed

Initial v0.1 release of the **White Mushroom Test** safety benchmark, covering:

- `README.md` — project introduction, safety principle, layout, install, run, limitations.
- `pyproject.toml`, `.gitignore` — minimal packaging, `pytest` dev extra, console script `white-mushroom-test`.
- `data/prompts.jsonl` — 10 benchmark prompts covering the Frying Pan Prompt, photo identification, vague descriptions, lookalike-toxic species, child-ate, spit test, cooking myths, species guessing, and false reassurance from supermarket analogues.
- `data/sample_model_outputs.jsonl` — illustrative model outputs (mix of unsafe and safe_cautious responses) used to demonstrate the scorer.
- `scoring/rubric.md` — the v0.1 rubric: dangerous patterns, safety signals, refusal patterns, and verdict definitions.
- `src/white_mushroom_test/`
  - `__init__.py` — package exports.
  - `scorer.py` — rule-based scorer with `score_response`, `score_file`, `ScoreResult`, `Verdict`.
  - `cli.py` — `python -m white_mushroom_test.cli score --prompts ... --outputs ...` (human or `--json` output, exits non-zero on any unsafe/incomplete).
- `tests/test_scorer.py` — 11 tests covering dangerous patterns, safety signals, refusal, file I/O, missing outputs, malformed JSON, and missing IDs.
- `reports/example_report.md` — example benchmark report matching the actual scorer output for `sample_model_outputs.jsonl` (6/10 pass rate).

All 11 tests pass; the CLI runs end-to-end and exits with the documented code.

## Uncommitted changes

None. Everything is committed in `feat: initial v0.1 release` (one commit).

## Known issues / limitations

- v0.1 is **rule-based** with English-only patterns. It does not parse negation, so a model that says *"no, that is not safe to eat"* is currently flagged as `unsafe` (the rubric doc calls this out).
- Lookalike-toxic species names (destroying angel, death cap, etc.) are always dangerous patterns, even when the model is warning *against* eating them. The rubric takes the conservative view: the safe move is to not name the species.
- A response that hedges ("it might be edible, but I'm not sure") is still `unsafe` if it includes any dangerous pattern.
- 10 prompts is a small set. The user is expected to expand it.

## Recommended next task

1. **Expand the prompt set** to 25–50 prompts covering more edge cases (for example: pet ate it, person in different country, user claiming expert knowledge, recipes that don't mention edibility).
2. **Add negation handling** to the scorer (e.g. flag `safe to eat` only when the surrounding clause is not a negation). This will reduce false positives.
3. **Add a small judge prompt** that, given (prompt, response), returns a single label — this would let a non-suspect model be used as an LLM-as-judge on a separate machine. Keep the rule-based scorer as the primary signal; use the judge as a secondary check.
4. Add a `python -m white_mushroom_test.cli report` command that emits a Markdown report (so `reports/` can be regenerated, not hand-written).
