# Handoff — White Mushroom Test v0.2 (image manifest pilot)

## What was completed

v0.2 adds an **image manifest** and a case generator that pairs every
image with every prompt. The release does **not** identify mushrooms,
does **not** publish edibility labels, and does **not** call any
model. The scorer is unchanged from v0.1.2.

### Files added

- **`data/images/README.md`** — schema docs and the safety principle
  for the image manifest.
- **`data/images/manifest.jsonl`** — 14 image rows (`wm_001` …
  `wm_014`). Every row sets
  `edibility_label_public: "withheld"` and
  `license: "private_test_only"`. Honest coverage across the allowed
  `image_quality` / `view` / `context` enum dimensions; the
  distribution of generated cases is:
  - `image_quality`: clear 50, medium 40, blurry 30, poor 20
  - `view`: side_view 30, cap_only 30, underside 20,
    full_stem_base 20, mixed_or_basket 20, unknown 10,
    cooking_context 10
  - `context`: grass 40, forest 30, yard 20, basket 20, unknown 10,
    frying_pan 10, plate 10
  No image files are committed. The case generator records
  `file_present: false` for every row by default.

- **`src/white_mushroom_test/generate_image_cases.py`** — schema
  constants, `ManifestError`, `ManifestRow` dataclass with strict
  `from_dict` validation, `load_manifest`, `load_prompts`, `build_case`,
  `generate_cases`, `write_cases`, and a CLI `main` that handles
  errors cleanly (exit 2 on validation failure, no partial output
  written).

- **`tests/test_generate_image_cases.py`** — 27 tests covering:
  - Row validation: well-formed rows pass; missing field, non-`withheld`
    edibility label, invalid enum values, non-bool
    `contains_multiple_mushrooms` are rejected.
  - All allowed enum values round-trip.
  - `load_manifest` skips blank lines, rejects invalid rows, rejects
    invalid JSON with a clear line-numbered message.
  - `load_prompts` requires `id` and `prompt`.
  - `build_case` records `file_present: true|false|None` correctly.
  - `generate_cases` produces `len(images) × len(prompts)` cases.
  - `--strict` raises on a missing image file before any output is
    written; without `--strict`, missing files are recorded.
  - `write_cases` writes one JSON object per line and creates parent
    dirs.
  - The shipped 14-row manifest validates cleanly and has 14 rows.

- **`data/generated/image_prompt_cases.jsonl`** — the 140 generated
  cases (14 images × 10 prompts). Re-generate with the CLI command in
  the README.

## Safety invariants preserved

- No mushroom identification, no edibility labels, no eating advice in
  any code, manifest, or generated output.
- The manifest validation rule
  `edibility_label_public == "withheld"` is a hard check
  (`ManifestError`) — a row that tries to publish an edibility label
  is rejected before any output is produced.
- The image files are not committed. The manifest is the only public
  artefact, and it does not contain species or edibility information.
- The generator does not call any model. The output is a list of
  cases for a future model runner; that runner is a separate task.

## Did the scorer or the v0.1.2 verdict taxonomy change?

No. v0.2 is purely an addition (image manifest + case generator).
The scorer, the public verdict set, and the test count for the
scorer (32) are unchanged from v0.1.2.

## Known issues / remaining limitations

- **No model runner yet.** The 140 cases in
  `data/generated/image_prompt_cases.jsonl` are inputs to a future
  vision-language model run. The runner is a separate task.
- **Pilot scope.** 14 images is a pilot, not a statistical sample.
  It is enough to spot gross failures in a model, not to draw
  conclusions. The manifest covers the allowed
  `image_quality` / `view` / `context` combinations honestly, not
  proportionally.
- **English-only prompts and labels.** The v0.1 prompts are English;
  the manifest `notes` are English.
- **Image files not committed.** The case generator records
  `file_present: false` for every shipped row. Bring your own
  images, or run with `--strict` once they are in place.

## Recommended next task

   `data/generated/image_prompt_cases.jsonl` and produces a JSONL of
   model outputs (one row per `case_id`, with `prompt_id`,
   `image_id`, `response`). The runner should be a separate
   module, not in `generate_image_cases.py`, and should record the
   model name and any per-case errors.
2. **Score the image run.** The existing `score` subcommand scores
   `prompt_id × response` rows; adapt it (or add a new subcommand)
   to score `case_id × response` rows so we can attribute failures
   to the image and the prompt independently.
3. **Expand the prompt set** to 25–30 prompts (cooking frames,
   photo-style with JSON descriptions, lookalike-toxic species, pet
   scenarios, misconception myths, adversarial frames, non-edibility
   framings). The v0.1.2 scorer is now stable enough to be the
   primary scoring signal for that expansion.
4. **Add a `compare` CLI subcommand** that runs the scorer across
   multiple output files and prints a side-by-side table.
