# Handoff — White Mushroom Test v0.4 (Ollama Vision Runner)

## What was completed

v0.4 ships a **local Ollama runner** that produces v0.3
`ModelOutputRow` files. The runner consumes the v0.2
generated (image × prompt) cases JSONL, pairs each case
with a local image file, sends the prompt and the image to a
local Ollama vision model, and writes one row per successful
case. Per-case errors (timeouts, refusals, missing images)
are recorded in a separate error JSONL and the run continues.

This is the **producer** side of the benchmark, joining the
v0.3.1 row-driven scorer (**consumer**). v0.3 and v0.3.1 are
otherwise unchanged: the runner only writes rows in the
existing schema, and the validator / scorer consume them
without modification.

### Files added

- **`src/white_mushroom_test/ollama_runner.py`** — pure
  stdlib runner. HTTP I/O via `urllib.request`, no
  third-party deps. Public surface:
  - `load_cases(path) -> list[dict]`
  - `resolve_image_path(image_dir, case) -> Path`
  - `encode_image_base64(path) -> str`
  - `build_ollama_payload(case, model, image_b64, temperature) -> dict`
  - `call_ollama(host, payload, timeout) -> str`
  - `make_model_output_row(case, model, response, latency_ms, host) -> dict`
  - `make_error_row(case, model, exc) -> dict`
  - `RunSummary` dataclass with `exit_code` property
  - `run_cases(...)` entry point
  - CLI `main(argv)` with the full 13-flag surface
- **`tests/test_ollama_runner.py`** — 15 unit tests. Every
  test that would otherwise hit a network is monkeypatched.
  Includes a project-level dry-run against the shipped
  140-case fixture.

### Files changed

- **`src/white_mushroom_test/cli.py`** — new
  `run-ollama` subcommand with all 13 flags, delegating to
  `ollama_runner.main`. Header bumped to `v0.4`.
- **`src/white_mushroom_test/__init__.py`** — exports
  `ollama_runner`, `__version__` bumped to `0.4.0`.
- **`README.md`** — v0.4 section with the safety
  principle, the dry-run → real run → validate → score
  workflow, the full flag table, and "what the runner does
  NOT do" (no identification, no eating advice, no API

- **`data/model_outputs/README.md`** — v0.4 addendum

- **`reports/example_report.md`** — v0.4 dry-run note

## Safety invariants preserved

- **No safety system prompt injected.** The case `prompt` is
  passed to Ollama verbatim. The benchmark observes the
  model's natural behaviour under the user prompt. The
  scorer evaluates the response afterwards. This is the
  headline safety principle of v0.4.
- **No third-party dependencies.** The runner uses
  `urllib.request` from the standard library. No `requests`,
  no `httpx`, no cloud SDKs.
- **No model API key.** Local Ollama only.
- **No image files committed.** Image files are
  user-supplied at runtime via `--image-dir`. The runner
  reads them from the user-supplied directory and never
  writes them anywhere.
- **No mushroom identification, no edibility labels, no
  eating advice** in any code, manifest, schema, README,
  or sample.
- **The runner does not modify the case file, the manifest,
  the prompts, or the scorer.** It only writes JSONL files
  into `data/model_outputs/` (and the error file path the
  user supplies).

## Known issues / remaining limitations

- **Image files are not committed.** A real model run
  requires the user to provide the 14 image files
  (`wm_001.jpg` … `wm_014.jpg`) in a local directory and
  point `--image-dir` at it. The shipped
  `data/images/manifest.jsonl` records filenames only; the
  files themselves are not in the repository.
- **No cloud-API runners yet.** Only Ollama is supported.
  An OpenAI / Anthropic runner would slot into the same
  `ModelOutputRow` schema.
- **No `compare` subcommand.** Scoring multiple output
  files side-by-side is still a manual process.
- **No `report` subcommand.** Markdown reports are still
  hand-edited (`reports/example_report.md`).
- **English-only.** Same as v0.1.2 / v0.3 / v0.3.1.
- **One row per case, no batching.** The runner sends one
  request per case. For 140 cases on a slow vision model,
  a full run is hours. A future version may add
  `--batch-size` / `--parallel` flags.

## Recommended next task

1. **Add a `compare` CLI subcommand** that scores multiple
   output files and prints a side-by-side table. With
   v0.3.1's row-driven scoring and the v0.4 runner, this
   becomes the natural way to compare, say, `gemma3:4b` vs
   `gemma3:12b` on the same case manifest.
2. **Add a `report` CLI subcommand** that emits a Markdown
   report from a scored output file, so
   `reports/example_report.md` can be regenerated, not
   hand-edited.
3. **Add per-image failure attribution** in the report
   generator: group results by `image_id` / `image_quality`
   / `view` / `context` to see whether models fail
   differently on different images.
4. **Expand the prompt set** to 25–30 prompts.
5. **Capture a real model run.** Run
   `run-ollama --model gemma3:4b` against the shipped
   140-case fixture with the user-supplied image files
   and commit the resulting `ModelOutputRow` JSONL to
   `data/model_outputs/`. This is the first end-to-end
   real-model run of the benchmark and will produce the
   first data-driven v0.4 report.
