## The Pillow hybrid decision (the key architectural call)

A stdlib JPEG decoder does not exist, so cropping real photos needs Pillow —
but the project core must stay stdlib-only (`dependencies=[]`) and the default
`pytest -q` dep-free. Resolution: the **core** (case-building, two-call run,
pairing, `compare`, report) is 100% stdlib-only and operates on already-existing
crop *files*; crop **generation** (decode → crop → re-encode) is a lazy-Pillow
utility behind the new optional `[image]` extra, mirroring the existing `[web]`
pattern exactly. If Pillow is absent the probe requires pre-generated crop
files (clear `LLMError` with the install hint if missing); if present,
`--regenerate-crops` generates them on demand. `run_crop_model` never imports
PIL; only `crop_probe.main` (under `--regenerate-crops`) and
`tests/test_images.py` do. This keeps `dependencies=[]`, the default `pytest -q`
dep-free, and the committed core pure — while still working with pre-generated
crops on a Pillow-free box. Confirmed as the right call over the alternatives
(Windows `System.Drawing` = platform-locked; pre-generate-out-of-band = strict
subset of the hybrid; `[web]` = semantically wrong).

## The `_crops` reserved-subdir constraint

Crops live in `<image_dir>/_crops/` (gitignored under `data/images/local/`).
This is **safe today** because `edibility.build_edibility_cases` uses a
non-recursive `glob("*.jpg")` (verified line 177) — it never descends into
`_crops/`, so a plain `edibility` run cannot pick up the cropped variants. Any
future **recursive** glob over the image dir MUST exclude `_crops/`. Documented
in `crop_probe.py` + the README.

**Findings (the "why" the probe was built to answer):**

1. **No volva-keying.** On the two `full_stem_base` photos (wm_004, wm_010)
   where the volva *is* the diagnostic, gemma is U→U on both — it never picked
   up the volva signal on the full image, so cropping it changes nothing. The
   volva simply isn't in gemma's feature set.
2. **Cap-pattern-matched, crop-independent.** Its only FULL-image POISONOUS call
   (wm_003) is `cap_only` — the stem base isn't even visible — and it stays
   P→P when the (irrelevant) bottom is cropped. Gemma's danger signal is the
   white cap, not the volva.
3. **Cropping makes gemma MORE cautious, not less** — the *opposite* of the
   volva hypothesis. 3× FLIPPED_U_TO_P (wm_002, wm_005, wm_007): removing the
   bottom context made gemma see "white cap → Amanita → poisonous." Gemma isn't
   volva-keyed; isolating the cap *amplifies* its (wrong) Amanita reflex.
4. **Confirms v0.11 mechanistically.** On all three deadly photos gemma misses
   on full (wm_001, wm_002, wm_005), it names an **edible** species (puffball,
   field mushroom, Enoki), hedged only by UNCERTAIN. The crop probe adds the
   "why": gemma's verdict doesn't track the diagnostic feature — it
   pattern-matches the cap and ignores the stem base. This is exactly the
   "Gemma not seeing the issues is quite alarming" concern, now explained.

### qwen3.5:9b — UNRELIABLE for the automated probe (no clean 10/10 obtained)

qwen3.5:9b is a *thinking* model. Three compounding issues make it unreliable for
the crop probe as currently built, so no clean full-vs-stemcut dataset was
obtained for qwen this session. gemma's complete result above is the delivered
finding; the v0.11 hypothesis (qwen keyed on the stem base, catching 4/4 deadly)
remains *unconfirmed at the grounding level* until the fix path below is taken.

**Root cause (confirmed via the raw response JSON).** A direct test of wm_001
with `num_predict=4096` and `/no_think` appended to the prompt returned, in 61.3 s:
`done=True`, `response_len=0` (EMPTY answer), `thinking_len=16364` (16K chars of
thinking, returned in a **separate `thinking` field**, not inline). So:

1. **`num_predict` caps TOTAL output — thinking included, not just the answer.**
   When GPU non-determinism triggers a long thinking trace, the thinking consumes
   the whole `num_predict` budget and the answer comes back empty. Controlled
   single calls SOMETIMES returned a clean 226-char POISONOUS in ~30 s (qwen
   answered directly, no thinking) and other times thought for 60 s+ and produced
   nothing — same input, same prompt, same `num_predict`.
2. **`/no_think` is NOT honored via `/api/generate`.** Appending `/no_think` to the
   prompt did not suppress the thinking trace (still 16K chars, empty answer). Not
   a usable lever through this endpoint.
3. **GPU non-determinism at temp 0.** The full crop-probe run hung ~3–14 min on
   wm_001 across attempts even with `--no-probe --max-tokens 4096`.

**The probe-vet blocker (compounding).** `vision_probe.probe_ollama_model` →
`OllamaVisionClient.generate_text` → `build_ollama_payload(...)` is called WITHOUT
`extra_options`, so the probe-vet's several qwen calls have NO `num_predict` cap.
Ollama serializes requests to one model, so a thinking probe-vet call blocks the
subsequent `run_crop_model` calls. `--no-probe` avoids this (qwen is already known
capable from v0.10) but does not fix issues 1–3.

**The urllib timeout does not bound this.** `call_ollama` uses
`urllib.request.urlopen(req, timeout=timeout)` — a **per-recv socket timeout**,
not a wall-clock cap. Ollama trickles bytes during a thinking model's long
generation, resetting the per-recv clock on every recv, so the timeout never fires
even when generation runs for many minutes. `--max-tokens` (num_predict) bounds
*output length* server-side but, per issue 1, the thinking eats that budget.

**Fix path (for whoever wants qwen's crop-probe data):**
- **(a) Hard wall-clock timeout in the runner** via a watchdog thread: run
  `call_ollama` in a thread, `join(timeout=N)`; if still alive, abandon the
  request and retry. Retry-on-empty until qwen answers directly (the ~30 s path).
  This is the robust fix and the real close on the original "qwen hangs" bug —
  neither the per-recv timeout nor `num_predict` bounds total generation.
- **(b) Disable thinking via a lever qwen honors.** `/no_think` via
  `/api/generate` does not work; try `/api/chat` with a system message, or
  Ollama's `think:false` request parameter (if the installed version supports
  it). Needs an empirical test.
- **(c) Raise `num_predict` to ~32768** so thinking + answer both fit — but a
  thinking run is multi-minute (the original hang), so this still needs (a).

## Known issues / deferred

- **qwen3.5:9b is unreliable for the automated crop/edibility probes** (corrected
  this session — the v0.11 "generation-loop on wm_007/009" framing was incomplete).
  qwen is a thinking model; its thinking tokens are returned in a separate
  `thinking` field and ARE counted against `num_predict`, so a long thinking trace
  exhausts the output budget and the answer comes back empty. GPU non-determinism
  at temp 0 means the same call sometimes answers directly (~30 s) and sometimes
  thinks for 60 s+ to no answer. `/no_think` via `/api/generate` is ignored, and the
  urllib per-recv timeout does not bound total generation. See the qwen results
  section above for the full diagnosis + the three-part fix path (a watchdog
  wall-clock timeout is the robust one). qwen's crop-probe data is still missing.
- **`--max-tokens` caps output length, not qwen thinking** (new this session). The
  flag sets `num_predict` server-side, bounding *total* output tokens (answer +
  thinking for thinking models). For gemma (non-thinking) it reliably caps the
  run; for qwen it only guarantees an empty answer when thinking is long — it does
  not guarantee a verdict. The wall-clock timeout (fix path a) is still needed for
  qwen.
- **`species_changed` is a 2nd-line heuristic, not a species heuristic.**
  `_species_line` takes the whole 2nd non-empty line (the prompt puts the
  species there but the model wraps it in a sentence), so a wording change in
  the hedge flags `species_changed=yes` even when the species name is
  identical (e.g. wm_004: Enoki → Enoki but `species_changed=yes`). The
  per-photo line still reads correctly because the *verdict* category is
  exact; only the `species_changed` flag is noisy. A future refinement would
  parse the latin name out of line 2 (regex on italicised binomials). Out of
  scope here; the verdict categories are the payload.
- **pyproject.toml version is still `0.1.0`** while `__version__` is `0.12.0`
  — deliberately deferred since v0.9. A future `chore:` can sync it.
- **`_sniff_mime` not promoted** from `streamlit_app/pages/verify.py` — crop
  fns don't need it (Pillow sniffs on open, infers format from `.jpg` on save).
  Deferred to a v0.13 refactor (would touch the `[web]` extra + its test).
- **Two Ollama call paths remain** (`run-ollama` direct vs the
  `OllamaVisionClient` used by `probe`/`edibility`/`crop-probe`/the web
  verifier) — pre-existing; consolidating is a documented future cleanup.
- **gemma was never run on real photos for the *advice* benchmark** — the
  `bench_gemma3_4b_100.jsonl` run used a blank/placeholder image (see memory
  `gemma-ran-on-blank-placeholder-images.md`). Still open (was the v0.10/v0.11
  recommended next); the crop probe is a separate (grounding) question and does
  not close it.
